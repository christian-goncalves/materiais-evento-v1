# Evidencia - Diagnostico tecnico das falhas 500

Data: 2026-04-03
Base URL: http://localhost:3010

## Sintoma inicial

- `PUT /api/pedidos` -> 500
- `POST /api/estoque/transferencia` -> 500
- `POST /api/estoque/venda-companheiro` -> 500
- `PATCH /api/financeiro/:id/repasse` roteando para `index.html`

## Causa raiz 1 - Roteamento repasse

Causa:
- Rewrite generico estava capturando rota de repasse dinamica no dev local.

Correcao:
- `vercel.json`: regra explicita para `/api/financeiro/([0-9]+)/repasse` -> `/api/financeiro/repasse?id=$1`.
- novo handler estatico em `api/financeiro/repasse.js`.

Evidencia:
- apos ajuste, endpoint responde JSON de negocio (404 para id inexistente), e nao mais `index.html`.

## Causa raiz 2 - 500 no PUT pedidos

Causa:
- endpoint de limpeza da Sheets API estava incorreto em `clearRanges`:
  - antes: `:batchClear`
  - correto: `/values:batchClear`

Impacto:
- `PUT /api/pedidos` falhava em runtime ao limpar faixas antes de gravar.

Correcao:
- `api/_sheets.js` atualizado para `await sheetsRequest('/values:batchClear', ...)`.

Evidencia:
- `PUT /api/pedidos` passou a retornar 200 com `{ ok: true }`.

## Estado dos endpoints apos correcao

- `PUT /api/pedidos` -> 200 (ok)
- `POST /api/estoque/transferencia` -> 400 `Estoque global insuficiente para transferencia` (regra de negocio)
- `POST /api/estoque/venda-companheiro` -> 400 `Saldo do companheiro insuficiente` (regra de negocio)
- `PATCH /api/financeiro/:id/repasse` -> 404 `Lancamento financeiro nao encontrado` quando id nao existe (regra de negocio)

## Conclusao objetiva

- Os 500 estruturais foram removidos para `PUT /api/pedidos` e rota de `repasse`.
- Os retornos atuais 400/404 restantes sao bloqueios de pre-condicao de dados (saldo e id valido), nao erro tecnico de infraestrutura.
